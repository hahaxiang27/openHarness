# Initializer Agent

This is the Initializer Agent. It should be executed only once for each project.

## Phase 1: Tech Stack Check

Verify the presence of the input/prd/tech-stack.md file.

If NOT found:
1. Write to .openharness/missing_info.json:
```json
{
  "missing_items": [{
    "id": "TECH_STACK_REQUIRED",
    "desc": "Please create `input/prd/tech-stack.md` to define the tech stack",
    "action_type": "human_action",
    "status": "pending"
  }]
}
```
2. Output: `--- INITIALIZATION BLOCKED: TECH_STACK_REQUIRED ---`
3. Exit cleanly

If found, continue to Phase 2.

---

## Phase 2: Git Setup

Generate the `.gitattributes` file:
```
* text=auto eol=lf
*.bat text eol=crlf
```

Append to `.gitignore` (create if not exists):
```
# SQL files generated by openHarness - should not be committed to the repository
*.sql
```

**Git Repository Check (IMPORTANT - MUST adhere to this specific order)**:

**Step 1: Check subdirectories for .git (MUST do this first!)**
```bash
# List all subdirectories and check for .git
ls -d */ | while read dir; do
  if [ -d "${dir}.git" ]; then
    echo "Git repo found in subdirectory: ${dir}"
    echo "STOP: Do NOT create parent git repository!"
    # Exit Phase 2 immediately
    return
  fi
done
```

**Step 2: Check if already in a git repository**
```bash
git rev-parse --git-dir 2>/dev/null
# If this succeeds, git already exists
# If this fails, proceed to Step 3
```

**Step 3: Check for .git in current directory**
```bash
if [ -d ".git" ]; then
  echo "Git already initialized in current directory"
  # Skip initialization
fi
```

**Step 4: Only initialize if ALL checks failed**
```bash
git init
git add .
git commit -m "Initial commit"
```

**CRITICAL RULE**: 
- If ANY subdirectory has `.git`, STOP immediately and DO NOT create parent git
- If current directory is already in a git repo, DO NOT reinitialize
- Only run `git init` if ALL above checks failed

## Phase 3: Feature List

**NOTE**: If the .openharness/feature_list.json file is already present, bypass this phase!

Check if feature_list.json exists:
```bash
if [ -f .openharness/feature_list.json ]; then
  echo "feature_list.json already exists, skipping feature generation"
  # Skip to Phase 4
fi
```

**Only create feature_list.json if it does NOT exist**:

**Step 1: Scan Tech Spec Files (Dynamic Discovery)**

Search the `input/techspec/` directory for files matching `tech-spec-*.md`

Extract category from filename:
- Remove the `tech-spec-` prefix and `.md` suffix
- Example: tech-spec-entity.md → entity
- Example: tech-spec-react-component.md → react-component
- Example: tech-spec-frontend-form.md → frontend-form

**No hardcoded whitelist** — ALL categories extracted from filenames are valid.

**Step 1.1: Read Scope from Each Tech Spec**

For each tech spec file, read its scope section (usually `## Scope`):
- Extract **file_patterns**: which source files this spec applies to (e.g., `*.tsx`, `*Service.java`)
- If no explicit scope section exists, infer applicability from the spec content

Build a tech_spec_registry in memory with: category, file_patterns, key_points, full_spec_file

**Step 2: Extract Key Points**

For each tech spec file, extract key_points (300-500 chars):
- Find the "Key Requirements" section (or `## 2.`)
- Extract items marked with `**MUST**`, `**FORBID**`, `**NORM**`
- Include code examples from the "Correct Examples" and "Incorrect Examples" sections

**Step 3: Product Module Identification**

Read input/PRD/*.md:
- Extract module names from headings
- Extract functions per module
- Identify function types (CRUD, query, batch, etc.)

**Extract PRD Requirements**:

Also extract the following:

| Extract Type | Pattern |
|-------------|---------|
| Table structure | CREATE TABLE, field list, table definition |
| Field constraints | field type, NOT NULL, PRIMARY KEY |
| Constraint words | must / forbid / force / not supported |
| Control list | buttons, dropdowns, table columns, input fields |

**Step 3.5: Risk Assessment**

After analyzing PRD, if any of the following risks are detected — contradictory logic, financial/fund safety, accounting/voucher errors, tax calculation risks, or batch data deletion/overwrite without safeguards — write to .openharness/missing_info.json with action_type="human_action" and STOP. Do not proceed to Step 4.

**Step 4: Feature Splitting (AI-Driven)**

For each product function identified from PRD:

1. Determine which categories from the tech_spec_registry are relevant
   to this function, based on:
   - The function's nature (what it does)
   - Each tech spec's scope/content (what it covers)
   - The AI's own judgment — no predefined mapping rules

2. Create one feature per relevant category

3. Determine dependencies between features based on:
   - Logical development order inferred by AI from context
   - The AI's understanding of the tech stack from input/prd/tech-stack.md
   - No hardcoded category chains

4. Assign tech_specs to each feature:
   - Primary spec: the spec matching the feature's own category
   - Related specs: other specs the AI judges relevant (e.g., a style spec
     for a component feature, a type-safety spec for any typed-language feature)
   - Each entry: { category, key_points, full_spec_file }

Each feature gets:
- tech_specs (array of specs, each with category, key_points, full_spec_file)
- dependencies (determined by AI based on logical order)

**Feature Splitting Rules**:

✅ DO:
- Split only explicit requirements from PRD
- Include implicit requirements (validation, pagination, error handling)
- Reference existing code patterns
- Extract table_design/constraints from PRD into prd_requirements
- Generate requirement_checklist: one item per field/control/constraint

❌ DON'T:
- Add "reserved for future" features
- Over-engineer (caching, monitoring unless required)
- Assume "might be needed" features
- Omit technical details from PRD (e.g., "JSON storage is forbidden")

**Feature Structure**:
```json
{
  "id": 1,
  "product_module": "module-name",
  "category": "entity",
  "description": "Entity class implementation for xxx",
  "implementation_detail": "Create XxxDO.java with required fields from PRD",
  "prd_requirements": {
    "table_design": "Table structure from PRD",
    "constraints": []
  },
  "requirement_checklist": [
    "Create required fields",
    "Add required annotations"
  ],
  "tech_specs": [
    {
      "category": "entity",
      "key_points": "Key rules extracted from tech-spec-entity.md",
      "full_spec_file": "input/techspec/tech-spec-entity.md"
    },
    {
      "category": "checkstyle",
      "key_points": "Key rules extracted from tech-spec-checkstyle.md",
      "full_spec_file": "input/techspec/tech-spec-checkstyle.md"
    }
  ],
  "dependencies": [5],
  "status": "pending"
}
```

**Mandatory final features** (id 990-991):
- 990: Run multi-layer tests (static analysis, unit test generation + execution, compilation check)
- 991: Code review

Create .openharness/feature_list.json with complete feature list.
